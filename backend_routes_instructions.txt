Backend — Endpoints faltantes para el panel admin
===================================================

Todos los endpoints requieren:  Authorization: Bearer <token>
Todos los errores devuelven:    { "detail": "mensaje" }

Se asume que ya existe la tabla `routes` con columnas (id, name, description)
y la tabla `route_places` con columnas (route_id, place_id, order).
(route_places ya existe — se hace cascade delete cuando se borra un place.)

---

## 1. GET /admin/routes

Lista todas las rutas ordenadas por nombre.

Respuesta (array):
[
  {
    "id": 1,
    "name": "Ruta histórica",
    "description": "Recorrido por el centro histórico",   // nullable
    "places": [
      { "id": 5, "name": "Catedral", "category": "lugares" },
      { "id": 12, "name": "Panteón Nacional", "category": "lugares" }
    ]
  },
  ...
]

Nota: `places` debe estar ordenado por `route_places.order` ASC.
En el listado solo se usa `places.length`, pero incluirlo evita una segunda
llamada cuando el admin navega al detalle.

---

## 2. GET /admin/routes/{id}

Ruta individual con sus lugares.

Respuesta (mismo shape que un ítem de la lista):
{
  "id": 1,
  "name": "Ruta histórica",
  "description": "Recorrido por el centro histórico",
  "places": [
    { "id": 5, "name": "Catedral", "category": "lugares" },
    { "id": 12, "name": "Panteón Nacional", "category": "lugares" }
  ]
}

Error 404:  { "detail": "Ruta no encontrada" }

---

## 3. PUT /admin/routes/{id}

Actualiza nombre y/o descripción. Solo se editan los campos enviados.

Body (JSON):
{
  "name": "Nuevo nombre",          // opcional, no puede ser vacío si se envía
  "description": "Texto o null",   // nullable
  "place_ids": [5, 12, 3]          // opcional — reemplaza la lista completa de lugares
}

Respuesta: { "ok": true }

Comportamiento de place_ids:
- Si se omite: los lugares no se tocan
- Si se envía (incluso vacío []): reemplaza todos los lugares de la ruta
- El orden del array define el order_index en la DB

Error 400 si no hay ningún campo: { "detail": "No hay campos para actualizar" }
Error 404 si no existe: { "detail": "Ruta no encontrada" }

---

## 4. DELETE /admin/routes/{id}

Elimina la ruta y sus filas en `route_places` (cascade o manual).
No elimina los places individuales.

Respuesta: { "ok": true }

Error 404: { "detail": "Ruta no encontrada" }

---

## Notas de implementación

- El campo `places` en GET se construye con un JOIN:
    SELECT p.id, p.name, p.category
    FROM route_places rp
    JOIN places p ON p.id = rp.place_id
    WHERE rp.route_id = %s
    ORDER BY rp.order ASC

- Si `route_places` no tiene columna `order`, devolver los lugares en orden de inserción
  (o agregar la columna con un default serial).

- Autenticación: mismo mecanismo JWT que el resto de /admin/* (Bearer token HS256).

---

## Diferencias entre la spec y la implementación real
(Leer antes de codear el frontend admin)

### 1. La columna en route_places se llama `order_index`, no `order`
  El documento dice `route_places.order` pero en la DB real es `order_index`.
  No afecta la API — el ordenamiento ya está aplicado en el backend.

### 2. Los GET devuelven dos campos extra: `is_preset` y `start_time`
  La tabla `routes` tiene columnas adicionales no mencionadas en la spec:
  - `is_preset` (boolean) — true si es una ruta predefinida de la app
  - `start_time` (string HH:MM:SS o null) — hora recomendada de inicio

  Ejemplo de respuesta real:
  {
    "id": 14,
    "name": "Centro Histórico",
    "description": "...",
    "is_preset": true,
    "start_time": "07:00:00",
    "places": [ ... ]
  }

  El PUT no permite editar estos campos (solo name y description).
  Si el admin necesita editar is_preset o start_time, hay que agregar
  esos campos al body del PUT.

### 3. DELETE usa CASCADE automático de la DB
  No hace falta enviar una petición extra para borrar los route_places —
  la FK tiene ON DELETE CASCADE, el backend solo borra la ruta y la DB
  limpia las filas relacionadas sola.

### 4. POST /admin/routes — crear ruta (no estaba en la spec original)

  Body (JSON):
  {
    "name": "Nueva ruta",          // requerido, no puede estar vacío
    "description": "Texto o null", // opcional
    "is_preset": false,            // opcional, default false
    "start_time": "08:00",         // opcional, formato HH:MM o null
    "place_ids": [5, 12, 3]        // opcional, orden de los lugares
  }

  Respuesta 201: { "id": 17 }

  Error 400 si name está vacío: { "detail": "El nombre no puede estar vacío" }

  Nota: place_ids es un array de IDs de lugares en el orden deseado.
  Si se omite o es [], la ruta se crea sin lugares.
